<h1 id="qunar-hotel-objective-c">Qunar Hotel Objective-C 风格指南<a href="#qunar-hotel-objective-c"></a></h1>&#x000A;<p>这份风格指南概括了纽约时报 iOS 团队的代码约定。</p>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">背景介绍<a href="#"></a></h2>&#x000A;<p>这里是苹果提供的一些风格指南，如果有什么东西没在这里提到，那么它可能在其中一个有详细的介绍：</p>&#x000A;&#x000A;<ul>&#x000A;<li><a href="http://developer.apple.com/library/mac/#documentation/Cocoa/Conceptual/ObjectiveC/Introduction/introObjectiveC.html">Objective-C 编程语言</a></li>&#x000A;<li><a href="https://developer.apple.com/library/mac/#documentation/Cocoa/Conceptual/CocoaFundamentals/Introduction/Introduction.html">Cocoa 基本原理指南</a></li>&#x000A;<li><a href="https://developer.apple.com/library/mac/#documentation/Cocoa/Conceptual/CodingGuidelines/CodingGuidelines.html">Cocoa 编码指南</a></li>&#x000A;<li><a href="http://developer.apple.com/library/ios/#documentation/iphone/conceptual/iphoneosprogrammingguide/Introduction/Introduction.html">iOS 应用编程指南</a></li>&#x000A;</ul>&#x000A;&#x000A;<p>本风格指南借鉴自 <a href="https://github.com/NYTimes/objective-c-style-guide">纽约时报 移动团队 Objective-C 风格指南</a> 及其中文版本 <a href="https://github.com/VincentSit/NYTimes-Objective-C-Style-Guide-ZH">纽约时报 移动团队 Objective-C 风格指南 中文版</a> 之后更新可参考这两个项目</p>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">目录<a href="#"></a></h2>&#x000A;<ul>&#x000A;<li><a href="#点语法">点语法</a></li>&#x000A;<li><a href="#间距">间距</a></li>&#x000A;<li><a href="#条件判断">条件判断</a>&#x000A;&#x000A;<ul>&#x000A;<li><a href="#三目运算符">三目运算符</a></li>&#x000A;</ul></li>&#x000A;<li><a href="#错误处理">错误处理</a></li>&#x000A;<li><a href="#方法">方法</a></li>&#x000A;<li><a href="#变量">变量</a></li>&#x000A;<li><a href="#命名">命名</a></li>&#x000A;<li><a href="#注释">注释</a></li>&#x000A;<li><a href="#init-和-dealloc">Init 和 Dealloc</a></li>&#x000A;<li><a href="#字面量">字面量</a></li>&#x000A;<li><a href="#CGRect-函数">CGRect 函数</a></li>&#x000A;<li><a href="#常量">常量</a></li>&#x000A;<li><a href="#枚举类型">枚举类型</a></li>&#x000A;<li><a href="#位掩码">位掩码</a></li>&#x000A;<li><a href="#头文件引用">头文件引用</a></li>&#x000A;<li><a href="#私有属性">私有属性</a></li>&#x000A;<li><a href="#图片命名">图片命名</a></li>&#x000A;<li><a href="#布尔">布尔</a></li>&#x000A;<li><a href="#单例">单例</a></li>&#x000A;<li><a href="#Xcode-工程">Xcode 工程</a></li>&#x000A;</ul>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">点语法<a href="#"></a></h2>&#x000A;<p>应该 <strong>始终</strong> 使用点语法来访问或者修改属性，访问其他实例时首选括号。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">view.backgroundColor = [UIColor orangeColor];&#x000A;[UIApplication sharedApplication].delegate;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">[view setBackgroundColor:[UIColor orangeColor]];&#x000A;UIApplication.sharedApplication.delegate;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">间距<a href="#"></a></h2>&#x000A;<ul>&#x000A;<li>一个缩进使用 4 个空格，永远不要使用制表符（tab）缩进。请确保在 Xcode 中设置了此偏好。</li>&#x000A;<li>方法的大括号和其他的大括号（<code>if</code>/<code>else</code>/<code>switch</code>/<code>while</code> 等等）始终和声明在同一行开始，在新的一行结束。</li>&#x000A;</ul>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if (user.isHappy) {&#x000A;// Do something&#x000A;}&#x000A;else {&#x000A;// Do something else&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<ul>&#x000A;<li>方法之间应该正好空一行，这有助于视觉清晰度和代码组织性。在方法中的功能块之间应该使用空白分开，但往往可能应该创建一个新的方法。</li>&#x000A;<li><code>@synthesize</code> 和 <code>@dynamic</code> 在实现中每个都应该占一个新行。</li>&#x000A;</ul>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">条件判断<a href="#"></a></h2>&#x000A;<p>条件判断主体部分应该始终使用大括号括住来防止<a>出错</a>，即使它可以不用大括号（例如它只需要一行）。这些错误包括添加第二行（代码）并希望它是 if 语句的一部分时。还有另外一种<a href="http://programmers.stackexchange.com/a/16530">更危险的</a>，当 if 语句里面的一行被注释掉，下一行就会在不经意间成为了这个 if 语句的一部分。此外，这种风格也更符合所有其他的条件判断，因此也更容易检查。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if (!error) {&#x000A;    return success;&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if (!error)&#x000A;    return success;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>或</p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if (!error) return success;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h3 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">三目运算符<a href="#"></a></h3>&#x000A;<p>三目运算符，? ，只有当它可以增加代码清晰度或整洁时才使用。单一的条件都应该优先考虑使用。多条件时通常使用 if 语句会更易懂，或者重构为实例变量。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">result = a &gt; b ? x : y;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">result = a &gt; b ? x = c &gt; d ? c : d : y;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">错误处理<a href="#"></a></h2>&#x000A;<p>当引用一个返回错误参数（error parameter）的方法时，应该针对返回值，而非错误变量。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">NSError *error;&#x000A;if (![self trySomethingWithError:&amp;error]) {&#x000A;    // 处理错误&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">NSError *error;&#x000A;[self trySomethingWithError:&amp;error];&#x000A;if (error) {&#x000A;    // 处理错误&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>一些苹果的 API 在成功的情况下会写一些垃圾值给错误参数（如果非空），所以针对错误变量可能会造成虚假结果（以及接下来的崩溃）。</p>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">方法<a href="#"></a></h2>&#x000A;<p>在方法签名中，在 -/+ 符号后应该有一个空格。方法片段之间也应该有一个空格。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">- (void)setExampleText:(NSString *)text image:(UIImage *)image;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">变量<a href="#"></a></h2>&#x000A;<p>变量名应该尽可能命名为描述性的。除了 <code>for()</code> 循环外，其他情况都应该避免使用单字母的变量名。&#x000A;星号表示指针属于变量，例如：<code>NSString *text</code> 不要写成 <code>NSString* text</code> 或者 <code>NSString * text</code> ，常量除外。&#x000A;尽量定义属性来代替直接使用实例变量。除了初始化方法（<code>init</code>， <code>initWithCoder:</code>，等）， <code>dealloc</code> 方法和自定义的 setters 和 getters 内部，应避免直接访问实例变量。更多有关在初始化方法和 dealloc 方法中使用访问器方法的信息，参见<a href="https://developer.apple.com/library/mac/documentation/Cocoa/Conceptual/MemoryMgmt/Articles/mmPractical.html#//apple_ref/doc/uid/TP40004447-SW6">这里</a>。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">@interface NYTSection: NSObject&#x000A;&#x000A;@property (nonatomic) NSString *headline;&#x000A;&#x000A;@end&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">@interface NYTSection : NSObject {&#x000A;    NSString *headline;&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h4 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">变量限定符<a href="#"></a></h4>&#x000A;<p>当涉及到<a>在 ARC 中被引入</a>变量限定符时，&#x000A;限定符 (<code>__strong</code>, <code>__weak</code>, <code>__unsafe_unretained</code>, <code>__autoreleasing</code>) 应该位于星号和变量名之间，如：<code>NSString * __weak text</code>。</p>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">命名<a href="#"></a></h2>&#x000A;<p>如非直接引用开源代码或第三方代码，默认类命名以 <code>QWH</code> 开头</p>&#x000A;&#x000A;<p>尽可能遵守苹果的命名约定，尤其那些涉及到<a href="https://developer.apple.com/library/mac/#documentation/Cocoa/Conceptual/MemoryMgmt/Articles/MemoryMgmt.html">内存管理规则</a>，（<a href="http://stackoverflow.com/a/2865194/340508">NARC</a>）的。</p>&#x000A;&#x000A;<p>长的和描述性的方法名和变量名都不错。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">UIButton *settingsButton;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">UIButton *setBut;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>类名和常量应该始终使用三个字母的前缀（例如 <code>NYT</code>），但 Core Data 实体名称可以省略。为了代码清晰，常量应该使用相关类的名字作为前缀并使用驼峰命名法。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">static const NSTimeInterval NYTArticleViewControllerNavigationFadeAnimationDuration = 0.3;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">static const NSTimeInterval fadetime = 1.7;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>属性和局部变量应该使用驼峰命名法并且首字母小写。</p>&#x000A;&#x000A;<p>为了保持一致，实例变量应该使用驼峰命名法命名，并且首字母小写，以下划线为前缀。这与 LLVM 自动合成的实例变量相一致。&#x000A;<strong>如果 LLVM 可以自动合成变量，那就让它自动合成。</strong></p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">@synthesize descriptiveVariableName = _descriptiveVariableName;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">id varnm;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">注释<a href="#"></a></h2>&#x000A;<p>当需要的时候，注释应该被用来解释 <strong>为什么</strong> 特定代码做了某些事情。所使用的任何注释必须保持最新否则就删除掉。</p>&#x000A;&#x000A;<p>通常应该避免一大块注释，代码就应该尽量作为自身的文档，只需要隔几行写几句说明。这并不适用于那些用来生成文档的注释。</p>&#x000A;&#x000A;<p>对于方法需要使用 Javadoc-like 格式进行注释，例如</p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">&#x000A;/**&#x000A; *  业务支付前校验通过，进行支付&#x000A; *&#x000A; *  @param payType          支付类型 和 页面信息  如：余额支付   支付宝支付  常用卡支付。。。 PayTypeVC  FillBankVC&#x000A; *&#x000A; *  @param resultType       ePayBusinessSuccess  ePayBusinessPaying&#x000A; */&#x000A;- (void)goToPay:(PayType *)payType andResultType:(PayBusinessResultType)resultType;&#x000A;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>上面的注释可添加 Xcode 插件 <code>VVDocumenter</code> 方便注释</p>&#x000A;&#x000A;<p>必须添加注释的情况包括：</p>&#x000A;&#x000A;<ol>&#x000A;<li><p>头文件中暴露给外部的接口需要添加严格的注释</p></li>&#x000A;<li><p>对于类实现代码中，对于非UI的业务逻辑方法需要添加对于功能说明的注释</p></li>&#x000A;</ol>&#x000A;<h2 id="init-dealloc">init 和 dealloc<a href="#init-dealloc"></a></h2>&#x000A;<p><code>dealloc</code> 方法应该放在实现文件的最上面，并且刚好在 <code>@synthesize</code> 和 <code>@dynamic</code> 语句的后面。在任何类中，<code>init</code> 都应该直接放在 <code>dealloc</code> 方法的下面。</p>&#x000A;&#x000A;<p><code>init</code> 方法的结构应该像这样：</p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">- (instancetype)init {&#x000A;    self = [super init]; // 或者调用指定的初始化方法&#x000A;    if (self) {&#x000A;        // Custom initialization&#x000A;    }&#x000A;&#x000A;    return self;&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">字面量<a href="#"></a></h2>&#x000A;<p>每当创建 <code>NSString</code>， <code>NSDictionary</code>， <code>NSArray</code>，和 <code>NSNumber</code> 类的不可变实例时，都应该使用字面量。要注意 <code>nil</code> 值不能传给 <code>NSArray</code> 和 <code>NSDictionary</code> 字面量，这样做会导致崩溃。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">NSArray *names = @[@&quot;Brian&quot;, @&quot;Matt&quot;, @&quot;Chris&quot;, @&quot;Alex&quot;, @&quot;Steve&quot;, @&quot;Paul&quot;];&#x000A;NSDictionary *productManagers = @{@&quot;iPhone&quot; : @&quot;Kate&quot;, @&quot;iPad&quot; : @&quot;Kamal&quot;, @&quot;Mobile Web&quot; : @&quot;Bill&quot;};&#x000A;NSNumber *shouldUseLiterals = @YES;&#x000A;NSNumber *buildingZIPCode = @10018;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">NSArray *names = [NSArray arrayWithObjects:@&quot;Brian&quot;, @&quot;Matt&quot;, @&quot;Chris&quot;, @&quot;Alex&quot;, @&quot;Steve&quot;, @&quot;Paul&quot;, nil];&#x000A;NSDictionary *productManagers = [NSDictionary dictionaryWithObjectsAndKeys: @&quot;Kate&quot;, @&quot;iPhone&quot;, @&quot;Kamal&quot;, @&quot;iPad&quot;, @&quot;Bill&quot;, @&quot;Mobile Web&quot;, nil];&#x000A;NSNumber *shouldUseLiterals = [NSNumber numberWithBool:YES];&#x000A;NSNumber *buildingZIPCode = [NSNumber numberWithInteger:10018];&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="cgrect">CGRect 函数<a href="#cgrect"></a></h2>&#x000A;<p>当访问一个 <code>CGRect</code> 的 <code>x</code>， <code>y</code>， <code>width</code>， <code>height</code> 时，应该使用<a href="http://developer.apple.com/library/ios/#documentation/graphicsimaging/reference/CGGeometry/Reference/reference.html">&lt;code&gt;CGGeometry&lt;/code&gt; 函数</a>代替直接访问结构体成员。苹果的 <code>CGGeometry</code> 参考中说到：</p>&#x000A;&#x000A;<blockquote>&#x000A;<p>All functions described in this reference that take CGRect data structures as inputs implicitly standardize those rectangles before calculating their results. For this reason, your applications should avoid directly reading and writing the data stored in the CGRect data structure. Instead, use the functions described here to manipulate rectangles and to retrieve their characteristics.</p>&#x000A;</blockquote>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">CGRect frame = self.view.frame;&#x000A;&#x000A;CGFloat x = CGRectGetMinX(frame);&#x000A;CGFloat y = CGRectGetMinY(frame);&#x000A;CGFloat width = CGRectGetWidth(frame);&#x000A;CGFloat height = CGRectGetHeight(frame);&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">CGRect frame = self.view.frame;&#x000A;&#x000A;CGFloat x = frame.origin.x;&#x000A;CGFloat y = frame.origin.y;&#x000A;CGFloat width = frame.size.width;&#x000A;CGFloat height = frame.size.height;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">常量<a href="#"></a></h2>&#x000A;<p>常量首选内联字符串字面量或数字，因为常量可以轻易重用并且可以快速改变而不需要查找和替换。常量应该声明为 <code>static</code> 常量而不是 <code>#define</code> ，除非非常明确地要当做宏来使用。在.m文件中使用到的UI常量可使用#define来进行定义，业务常量必须使用 static 来进行定义</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">static NSString * const NYTAboutViewControllerCompanyName = @&quot;The New York Times Company&quot;;&#x000A;&#x000A;#define thumbnailHeight = 50.0;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">#define CompanyName @&quot;The New York Times Company&quot;&#x000A;&#x000A;#define thumbnailHeight 2&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">枚举类型<a href="#"></a></h2>&#x000A;<p>当使用 <code>enum</code> 时，建议使用新的基础类型规范，因为它具有更强的类型检查和代码补全功能。现在 SDK 包含了一个宏来鼓励使用使用新的基础类型 - <code>NS_ENUM()</code></p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">typedef NS_ENUM(NSInteger, NYTAdRequestState) {&#x000A;    NYTAdRequestStateInactive,&#x000A;    NYTAdRequestStateLoading&#x000A;};&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">位掩码<a href="#"></a></h2>&#x000A;<p>当用到位掩码时，使用 <code>NS_OPTIONS</code> 宏。</p>&#x000A;&#x000A;<p><strong>举例：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">&#x000A;typedef NS_OPTIONS(NSUInteger, NYTAdCategory) {&#x000A;NYTAdCategoryAutos      = 1 &lt;&lt; 0,&#x000A;NYTAdCategoryJobs       = 1 &lt;&lt; 1,&#x000A;NYTAdCategoryRealState  = 1 &lt;&lt; 2,&#x000A;NYTAdCategoryTechnology = 1 &lt;&lt; 3&#x000A;};&#x000A;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">头文件引用<a href="#"></a></h2>&#x000A;<p>在头文件中引用其他类使用 @class 进行引用</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">&#x000A;#import &quot;LoadNaviVC.h&quot;&#x000A;&#x000A;@class FilterInfo;&#x000A;@class HSearchParam;&#x000A;&#x000A;@interface HHotelListVC : LoadNaviVC &lt;LocationDelegate, NetworkPtc, SearchBarDelgt, TabBarPtc, MFSideMenuPtc, SortViewDelegate, HFilterDelegate, DateVCPtc, HKeywordDelegate, FListenVoiceDownCloseDelgt, HHotelListTableDelgt, HHotelListMapDelgt, HCityVCDelegate&gt;&#x000A;&#x000A;@property (nonatomic, strong) FilterInfo                *filterInfoFilter;                      // 过滤条件&#x000A;@property (nonatomic, strong) FilterInfo                *filterInfoFilterTmp;                   // 临时过滤条件&#x000A;@property (nonatomic, strong) HSearchParam              *hSearchParam;                          // 酒店搜索请求参数&#x000A;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>在类实现中，如果使用到外部类的方法则需要用 @import 引用该类的头文件，否则不需要进行引用</p>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">私有属性<a href="#"></a></h2>&#x000A;<p>私有属性应该声明在类实现文件的延展（匿名的类目）中。有名字的类目（例如 <code>NYTPrivate</code> 或 <code>private</code>）永远都不应该使用，除非要扩展其他类。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">@interface NYTAdvertisement ()&#x000A;&#x000A;@property (nonatomic, strong) GADBannerView *googleAdView;&#x000A;@property (nonatomic, strong) ADBannerView *iAdView;&#x000A;@property (nonatomic, strong) UIWebView *adXWebView;&#x000A;&#x000A;@end&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">图片命名<a href="#"></a></h2>&#x000A;<p>图片名称应该被统一命名以保持组织的完整。它们应该被命名为一个说明它们用途的驼峰式字符串，其次是自定义类或属性的无前缀名字（如果有的话），然后进一步说明颜色 和/或 展示位置，最后是它们的状态。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<ul>&#x000A;<li><code>RefreshBarButtonItem</code> / <code>RefreshBarButtonItem@2x</code> 和 <code>RefreshBarButtonItemSelected</code> / <code>RefreshBarButtonItemSelected@2x</code></li>&#x000A;<li><code>ArticleNavigationBarWhite</code> / <code>ArticleNavigationBarWhite@2x</code> 和 <code>ArticleNavigationBarBlackSelected</code> / <code>ArticleNavigationBarBlackSelected@2x</code>.</li>&#x000A;</ul>&#x000A;&#x000A;<p>图片目录中被用于类似目的的图片应归入各自的组中。</p>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">布尔<a href="#"></a></h2>&#x000A;<p>因为 <code>nil</code> 解析为 <code>NO</code>，所以没有必要在条件中与它进行比较。永远不要直接和 <code>YES</code> 进行比较，因为 <code>YES</code> 被定义为 1，而 <code>BOOL</code> 可以多达 8 位。</p>&#x000A;&#x000A;<p>这使得整个文件有更多的一致性和更大的视觉清晰度。</p>&#x000A;&#x000A;<p><strong>推荐：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if (!someObject) {&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if (someObject == nil) {&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<hr>&#x000A;&#x000A;<p><strong>对于 <code>BOOL</code> 来说, 这有两种用法:</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if (isAwesome)&#x000A;if (![someObject boolValue])&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p><strong>反对：</strong></p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">if ([someObject boolValue] == NO)&#x000A;if (isAwesome == YES) // 永远别这么做&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<hr>&#x000A;&#x000A;<p>如果一个 <code>BOOL</code> 属性名称是一个形容词，属性可以省略 “is” 前缀（不做强制要求，但推荐为 get 访问器指定一个惯用的名字), 例如：</p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">@property (assign, getter=isEditable) BOOL editable;&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>内容和例子来自 <a href="https://developer.apple.com/library/mac/#documentation/Cocoa/Conceptual/CodingGuidelines/Articles/NamingIvarsAndTypes.html#//apple_ref/doc/uid/20001284-BAJGIIJE">Cocoa 命名指南</a> 。</p>&#x000A;<h2 id="/mobile_hotel_ios/qunar-objective-c-style-guide/blob/master/README.md#">单例<a href="#"></a></h2>&#x000A;<p>单例对象应该使用线程安全的模式创建共享的实例。</p>&#x000A;&#x000A;<div class="highlighted-data white">&#x000A;  <div class="highlight">&#x000A;    <pre><code class="objc">+ (instancetype)sharedInstance {&#x000A;   static id sharedInstance = nil;&#x000A;&#x000A;   static dispatch_once_t onceToken;&#x000A;   dispatch_once(&amp;onceToken, ^{&#x000A;      sharedInstance = [[self alloc] init];&#x000A;   });&#x000A;&#x000A;   return sharedInstance;&#x000A;}&#x000A;</code></pre>&#x000A;  </div>&#x000A;</div>&#x000A;&#x000A;&#x000A;<p>这将会预防<a href="http://cocoasamurai.blogspot.com/2011/04/singletons-your-doing-them-wrong.html">有时可能产生的许多崩溃</a>。</p>&#x000A;<h2 id="xcode">Xcode 工程<a href="#xcode"></a></h2>&#x000A;<p>为了避免文件杂乱，物理文件应该保持和 Xcode 项目文件同步。Xcode 创建的任何组（group）都必须在文件系统有相应的映射。为了更清晰，代码不仅应该按照类型进行分组，也可以根据功能进行分组。</p>&#x000A;&#x000A;<p>如果可以的话，尽可能一直打开 target Build Settings 中 &quot;Treat Warnings as Errors&quot; 以及一些<a href="http://boredzo.org/blog/archives/2009-11-07/warnings">额外的警告</a>。如果你需要忽略指定的警告,使用 <a href="http://clang.llvm.org/docs/UsersManual.html#controlling-diagnostics-via-pragmas">Clang 的编译特性</a> 。</p>&#x000A;<h1 id="objective-c">其他 Objective-C 风格指南<a href="#objective-c"></a></h1>&#x000A;<p>如果感觉我们的不太符合你的口味，可以看看下面的风格指南：</p>&#x000A;&#x000A;<ul>&#x000A;<li><a href="http://google-styleguide.googlecode.com/svn/trunk/objcguide.xml">Google</a></li>&#x000A;<li><a href="https://github.com/github/objective-c-conventions">GitHub</a></li>&#x000A;<li><a href="https://trac.adium.im/wiki/CodingStyle">Adium</a></li>&#x000A;<li><a href="https://gist.github.com/soffes/812796">Sam Soffes</a></li>&#x000A;<li><a href="http://cocoadevcentral.com/articles/000082.php">CocoaDevCentral</a></li>&#x000A;<li><a href="http://lukeredpath.co.uk/blog/my-objective-c-style-guide.html">Luke Redpath</a></li>&#x000A;<li><a href="http://www.cimgf.com/zds-code-style-guide/">Marcus Zarra</a></li>&#x000A;</ul>&#x000A;</div>
